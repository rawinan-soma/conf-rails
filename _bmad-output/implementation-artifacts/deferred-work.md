## Deferred from: code review of story-1.3-oidc-authentication-sessions (2026-06-19)

- Globally-unique `email` index can block legitimate logins [db/migrate/20260619000001_create_users.rb]: The unique index on `users.email` plus `validates :email, uniqueness:` means two distinct IdP subjects (different `uid`) that resolve to the same email (re-provisioned account, shared mailbox/alias) cannot both authenticate — the second login's create fails. The natural identity key is `(provider, uid)`; email uniqueness adds a login-failure mode. Deferred because the unique email index is explicitly mandated by Story 1.3 Task 1 ("Add unique index on `email`"). The controller now handles the resulting invalid/non-unique record gracefully (redirects to the failure flow instead of 500), so this is no longer a crash — but the underlying design constraint should be revisited if duplicate-email IdP scenarios are in scope. Reconsider whether email should be non-unique (or unique-scoped) when profile/identity work lands in Story 1.5.
